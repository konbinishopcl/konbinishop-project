# Phase 1 · Plan 01-01: Módulo `events` de la API

**One-liner:** Construir el módulo `events` en la API NestJS — listado público de eventos
aprobados, detalle por slug, CRUD para el organizador, listado propio, y endpoints de
moderación (aprobar/rechazar) protegidos por rol.

## Contexto

La API solo tiene `auth` y `users`. Los eventos viven en el `schema.prisma` (modelo `Event`
con componentes `EventPrice`, `EventDate`, `EventSocialLink`, `EventVideo` y relaciones a
`Region`, `Commune`, `Category`, `User`) pero no hay endpoints. El website usa data mock.
Este plan expone los eventos por HTTP — es el prerequisito de las fases 2–5.

No se requieren cambios de schema: el modelo `Event` ya existe.

## Endpoints

| Método | Ruta | Acceso | Descripción |
|--------|------|--------|-------------|
| GET | `/api/events` | público | Lista eventos aprobados, paginado; filtros `q`, `category`, `region` |
| GET | `/api/events/admin` | ADMIN+ | Lista todos los eventos (cualquier estado), con organizador |
| GET | `/api/events/mine` | autenticado | Eventos del usuario actual |
| GET | `/api/events/:slug` | público | Detalle de un evento aprobado |
| POST | `/api/events` | autenticado | Crea un evento → queda pendiente de moderación |
| PATCH | `/api/events/:id` | dueño o admin | Edita un evento |
| DELETE | `/api/events/:id` | dueño o admin | Elimina un evento |
| PATCH | `/api/events/:id/approve` | ADMIN+ | Aprueba — pasa a visible al público |
| PATCH | `/api/events/:id/reject` | ADMIN+ | Rechaza con motivo |

## Tareas

1. **DTOs** — `create-event.dto.ts` (con DTOs anidados para prices/dates/links),
   `update-event.dto.ts` (todo opcional), `query-events.dto.ts`, `reject-event.dto.ts`.
2. **`events.service.ts`** — lógica Prisma: listado público paginado, detalle, mine, admin,
   create (slug único + `isApproved=false`), update/delete (chequeo dueño-o-admin),
   approve/reject (setea flags + `approvedBy`/`rejectedBy`).
3. **`events.controller.ts` + `events.module.ts`** — rutas con guards; registrar el módulo
   en `app.module.ts`.
4. **Compilar y verificar** — `nest build` limpio; smoke test de los endpoints.

## Decisiones

- Un evento creado por un organizador queda `isApproved=false, isRejected=false`.
- `update`/`delete`: permitido al dueño (`userId`) o a un `ADMIN`/`SUPER_ADMIN`.
- `approve` limpia `rejectedReason` y `rejectedBy`; `reject` exige un `reason`.
- Las taxonomías (regiones, comunas, categorías, etc.) y la subida de imágenes van en los
  planes 01-02 y 01-03.

## Verificación

- `nest build` compila sin errores
- `GET /api/events` devuelve solo eventos aprobados, paginado
- `GET /api/events/:slug` devuelve el evento con componentes y relaciones
- `POST /api/events` con token de organizador crea el evento con `isApproved=false`
- `PATCH /api/events/:id/approve` con token admin lo vuelve visible; sin token → `401`;
  con token `AUTHENTICATED` → `403`
- Un organizador no puede editar el evento de otro (`403`)
