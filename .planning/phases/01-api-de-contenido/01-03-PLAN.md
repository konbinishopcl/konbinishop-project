# Phase 1 · Plan 01-03: Endpoint de subida de imágenes

**One-liner:** Endpoint `POST` autenticado que recibe una imagen, la guarda en
`apps/api/uploads/` y devuelve su URL pública — la última pieza de la API de contenido.

## Contexto

El almacenamiento ya está decidido y operativo: disco local en `apps/api/uploads/`, servido
en `/uploads` (hecho en 01-02 junto con el seed). Falta solo el endpoint que recibe el
archivo. Con esto el organizador (Phase 3) podrá subir banner/poster/galería de sus eventos.

## Endpoint

| Método | Ruta | Acceso | Descripción |
|--------|------|--------|-------------|
| POST | `/api/upload` | autenticado | Recibe `file` (multipart), guarda la imagen y devuelve `{ url }` |

## Tareas

1. **Módulo `uploads`** — `uploads.controller.ts` (POST con `FileInterceptor`),
   `uploads.service.ts` (valida tipo/tamaño, escribe el archivo, devuelve la URL),
   `uploads.module.ts`.
2. **Registrar** `UploadsModule` en `app.module.ts`.
3. **Normalizar** la ruta de `uploads/` — `main.ts` y el service resuelven el mismo
   directorio vía `process.cwd()`.
4. **Compilar y verificar** — `nest build` limpio + smoke test del POST.

## Decisiones

- `FileInterceptor` de `@nestjs/platform-express` con almacenamiento en memoria; el service
  escribe el buffer a disco con `fs`. Evita importar `multer` directamente (es dependencia
  transitiva, no resoluble con pnpm) y no agrega dependencias nuevas.
- Formatos permitidos: JPG, PNG, WebP. Límite de tamaño: 5 MB.
- Nombre de archivo único: `timestamp-random.ext`. Se devuelve la ruta relativa `/uploads/…`,
  igual que las imágenes del seed.
- El endpoint solo exige sesión (`JwtAuthGuard`), sin rol específico — los organizadores
  suben imágenes de sus eventos.

## Verificación

- `nest build` compila sin errores
- `POST /api/upload` sin token → `401`
- `POST /api/upload` con token + imagen → `200` con `{ url: "/uploads/…" }`; la URL devuelta
  responde `200` al pedirla
- `POST /api/upload` con un archivo no-imagen → `400`
