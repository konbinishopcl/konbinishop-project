**File 1: `apps/website/server/utils/recaptcha.ts`**

Copy verbatim from waldo-project (`/home/gab/Code/waldo-project/apps/website/server/utils/recaptcha.ts`). The file contains:

```typescript
import { createError } from "h3";

const RECAPTCHA_PROTECTED_METHODS = ["POST", "PUT", "DELETE"];

export async function verifyRecaptchaToken(
  token: string | null | undefined,
  secretKey: string,
): Promise<void> {
  if (!token || !token.trim()) {
    throw createError({
      statusCode: 400,
      statusMessage: "reCAPTCHA token is required",
    });
  }

  const result = await $fetch<{
    success: boolean;
    score: number;
    "error-codes"?: string[];
  }>("https://www.google.com/recaptcha/api/siteverify", {
    method: "POST",
    body: new URLSearchParams({
      secret: secretKey,
      response: token,
    }).toString(),
    headers: { "Content-Type": "application/x-www-form-urlencoded" },
  });

  if (!result.success || result.score <= 0.5) {
    console.warn(
      `[recaptcha] Verification failed. success=${result.success}, score=${result.score ?? "n/a"}, error-codes=${(result["error-codes"] ?? []).join(",")}`,
    );
    throw createError({
      statusCode: 400,
      statusMessage: "reCAPTCHA verification failed. Please try again.",
    });
  }
}

export function isRecaptchaProtectedRoute(
  _fullPath: string,
  method: string,
): boolean {
  return RECAPTCHA_PROTECTED_METHODS.includes(method.toUpperCase());
}
```

IMPORTANT: Adapt formatting to match konbini website conventions (no semicolons, single quotes, per `apps/website/.prettierrc`).

**File 2: `apps/website/server/api/[...].ts`**

Adapt from waldo-project but REMOVE the OAuth routes section (konbini does not use OAuth). The file:

```typescript
import {
  verifyRecaptchaToken,
  isRecaptchaProtectedRoute,
} from '../utils/recaptcha'

export default defineEventHandler(async (event) => {
  const config = useRuntimeConfig(event)
  const apiUrl = process.env.API_URL || 'http://localhost:1337'

  // Get the path after /api/
  const fullPath = event.node.req.url?.replace('/api/', '') || ''

  // reCAPTCHA validation for protected routes (POST/PUT/DELETE)
  if (isRecaptchaProtectedRoute(fullPath, event.method ?? '')) {
    const recaptchaToken = getHeader(event, 'x-recaptcha-token')
    await verifyRecaptchaToken(
      recaptchaToken,
      config.recaptchaSecretKey as string,
    )
  }

  // Build the target URL
  const targetUrl = `${apiUrl}/api/${fullPath}`

  // Forward only whitelisted headers
  const headers: Record<string, string> = {}

  const authHeader = getHeader(event, 'authorization')
  if (authHeader) {
    headers['Authorization'] = authHeader
  }

  const contentType = getHeader(event, 'content-type')
  if (contentType) {
    headers['Content-Type'] = contentType
  }

  const cookie = getHeader(event, 'cookie')
  if (cookie) {
    headers['Cookie'] = cookie
  }

  // Forward the request to Strapi
  return proxyRequest(event, targetUrl, { headers })
})
```

Key differences from waldo:
- No OAuth routes (removed entirely)
- Formatting follows konbini website conventions (no semicolons)
- Uses `config.recaptchaSecretKey` from runtimeConfig

**Change 2: Update @nuxtjs/strapi URL for CLIENT-SIDE requests only**

The `strapi.url` config controls where `@nuxtjs/strapi` sends requests. For CLIENT-SIDE (browser), it must go through the Nuxt proxy at the same origin. For SERVER-SIDE (SSR), it should go directly to Strapi to avoid loopback (see Research Pitfall 4).

`@nuxtjs/strapi` v2 with Nuxt 4 uses `runtimeConfig` for URL resolution. The `strapi.url` in the module config sets the default. SSR requests should go directly to Strapi via `API_URL`, while client requests should go through the proxy via relative URL.

Update the strapi config:

```typescript
strapi: {
  url: process.env.API_URL || 'http://localhost:1337',
  prefix: '/api',
  version: 'v4',
  cookie: {
    path: '/',
    maxAge: 86400,
  },
  cookieName: 'strapi_jwt',
},
```

IMPORTANT: Keep `strapi.url` pointing at Strapi directly (NOT at localhost:3000). The `@nuxtjs/strapi` module handles SSR by calling Strapi directly from the server. Client-side requests from the browser will go through the proxy because Vue components use `useStrapiClient` which builds URLs relative to the page origin when running in the browser. The catch-all `server/api/[...].ts` intercepts these requests.

WAIT -- this is incorrect. `@nuxtjs/strapi` sends requests to the configured `url` from BOTH server and client. On the client, it will send to `http://localhost:1337` directly, which defeats the purpose.

The correct approach per the waldo-project pattern: set `strapi.url` differently for server vs client using `runtimeConfig`. However, `@nuxtjs/strapi` v2 uses its own `strapi` config key, not `runtimeConfig`.

The SAFEST approach for this phase: keep `strapi.url` pointing at Strapi for now AND rely on the catch-all proxy to intercept any browser-side `/api/` calls naturally. Since `@nuxtjs/strapi` on the client constructs absolute URLs to `http://localhost:1337/api/events`, the browser will still call Strapi directly.

To actually fix this, we need to change `strapi.url` for the browser environment. The cleanest way:

```typescript
strapi: {
  url: process.env.STRAPI_URL || 'http://localhost:1337',  // Server-side (SSR): direct to Strapi
  prefix: '/api',
  version: 'v4',
  cookie: {
    path: '/',
    maxAge: 86400,
  },
  cookieName: 'strapi_jwt',
},
```

AND add a Nuxt plugin that overrides the Strapi URL on the client side:

Create `apps/website/plugins/strapi-proxy.client.ts`:

```typescript
export default defineNuxtPlugin(() => {
  const strapi = useStrapiClient()
  // No plugin override needed -- instead we override via runtimeConfig

```

Actually, the simplest and correct approach: `@nuxtjs/strapi` v2 for Nuxt 4 reads the URL from `runtimeConfig.public.strapi.url` at runtime. We can set this in `nuxt.config.ts`:

```typescript
runtimeConfig: {
  strapi: {
    url: process.env.API_URL || 'http://localhost:1337',  // server-side SSR: direct
  },
  public: {

      url: process.env.BASE_URL || 'http://localhost:3000',  // client-side: through proxy
    },
    // ... other public config
  },
},
```

BUT this may conflict with the `strapi` module config key. Test carefully.

**FINAL DECISION:** Use environment-based override. Set the strapi module config URL to point at the Nuxt origin for client, and use `runtimeConfig.strapi.url` for the server-side override:

The catch-all proxy `server/api/[...].ts` already reads `process.env.API_URL` directly, so SSR requests from `@nuxtjs/strapi` that hit `localhost:3000/api/*` will be caught by the proxy and forwarded to Strapi. This creates a loopback for SSR (Nuxt server calls itself), which is suboptimal but functional and safe. The Research flags this as MEDIUM confidence -- it works in waldo-project this way.
