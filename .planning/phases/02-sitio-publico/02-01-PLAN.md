# Phase 2 · Plan 02-01: Capa de datos + home con eventos reales

**One-liner:** Construir la capa de datos del website (cliente HTTP de eventos y catálogo) y
conectar la home para que muestre eventos reales de la API en vez de `lib/data.ts`.

## Contexto

La API de contenido (Phase 1) ya está completa. El website sigue 100% en mock (`lib/data.ts`).
Este plan arma el puente — tipos, fetchers y mappers — y lo estrena en la home. Las demás
vistas (categorías, detalle, checkout) van en los planes siguientes de Phase 2.

## Tareas

1. **`lib/api.ts`** — agregar:
   - Tipos de la API: `ApiEvent`, `ApiEventList`, `ApiCategory`, `ApiRegion`, `ApiCommune`.
   - Fetchers: `api.events(query?)`, `api.event(slug)`, `api.categories()`, `api.regions()`,
     `api.communes(region?)`.
   - `imageUrl(path)` — antepone el origen de la API a las rutas `/uploads/...`.
   - `toEventItem(ApiEvent)` — mapea el evento de la API al shape `EventItem` que usan las
     cards; formatea fecha y calcula el precio mínimo.
2. **`lib/data.ts`** — `EventItem` gana `slug` y `catSlug`; los campos solo-maqueta
   (`ja`, `art`, `stamp`) pasan a opcionales. Se agrega `slug` a los 12 eventos mock (siguen
   alimentando las vistas aún no conectadas).
3. **`EventCard.tsx`** — enlazar por `slug` (`/evento/<slug>`).
4. **`(site)/page.tsx`** — server component `async`: trae eventos y categorías de la API y
   arma el rail "Destacados" + un rail por categoría con eventos.

## Decisiones

- Las imágenes vienen como ruta relativa (`/uploads/...`); `imageUrl()` antepone el origen.
- La home es server component (`dynamic = "force-dynamic"`); si la API no responde, degrada
  a una home sin rails en vez de romper.
- El `HeroBlock` queda con datos mock por ahora — se conecta en un plan posterior (la API de
  heroes existe, pero sus imágenes de seed son placeholders).
- Detalle de evento por slug; el route `evento/[id]` se renombra a `[slug]` en el plan 02-03.

## Verificación

- `pnpm build` del website compila sin errores de tipos
- La home renderiza el rail "Destacados" + un rail por cada categoría con eventos, usando
  datos reales de la API (eventos del seed)
- Las cards enlazan a `/evento/<slug>`
