# Phase 2 · Plan 02-02: Categorías reales

**One-liner:** Conectar el nav de categorías del `Header` y la página `/categoria/[cat]` a la
API — hoy ambos usan las categorías mock (`cine/conciertos/...`), distintas de las reales.

## Contexto

Tras 02-01 la home ya usa datos reales, pero el `Header` todavía navega con las categorías
mock y `/categoria/[cat]` lista eventos mock. Las categorías reales (del seed) son
`musica`, `teatro`, `gastronomia`, `anime-y-manga`, `videojuegos`.

## Tareas

1. **`(site)/layout.tsx`** — server component `async`: trae las categorías de la API y se
   las pasa al `Header`. Si la API falla, pasa lista vacía.
2. **`Header.tsx`** — recibe `categories` por prop y arma el nav (botón "Inicio" fijo + una
   entrada por categoría real → `/categoria/<slug>`). Deja de importar `CATEGORIES` mock.
3. **`/categoria/[cat]/page.tsx`** — pasa de client component a server component `async`:
   resuelve la categoría por slug, trae sus eventos con `api.events({ category })` y los
   renderiza con `EventCard`. Estado vacío cuando no hay eventos.

## Decisiones

- El nav del `Header` lleva un "Inicio" fijo + las categorías de la API.
- Los chips de filtro de la página de categoría quedan estáticos (placeholder); los filtros
  reales son trabajo de Phase 5 (Búsqueda).
- Se quita el texto japonés decorativo de la cabecera de categoría (la API no provee ese
  dato) y el copy deja de mencionar "compra tu entrada" (alineado con events-only).

## Verificación

- `pnpm build` del website compila sin errores
- El nav del Header muestra las categorías reales y navega a `/categoria/<slug>`
- `/categoria/musica` (u otra) lista los eventos reales de esa categoría
- Una categoría sin eventos muestra el estado vacío
