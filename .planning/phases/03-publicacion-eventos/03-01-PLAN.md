# Phase 3 · Plan 03-01: Formulario /crear conectado a la API

**One-liner:** Conectar el formulario `/crear` (3 pasos) a `POST /events` — con sesión
requerida, catálogos reales y envío del evento, que queda pendiente de moderación.

## Contexto

`/crear` es un formulario de 3 pasos con estado puramente local; al "Publicar" solo hace un
`alert()`. La API ya tiene `POST /events` (Phase 1) y los catálogos. Este plan lo conecta de
verdad. Las imágenes se suben en el plan 03-02; el panel `/cuenta` en el 03-03.

## Tareas

1. **`lib/api.ts`** — tipo `CreateEventInput` + `api.createEvent(body, token)` → `POST /events`.
2. **`/crear/page.tsx`**:
   - **Auth gate**: sin sesión → redirige a `/login`.
   - **Catálogos reales**: el select de categoría usa `api.categories()`; región y comuna
     usan `api.regions()` / `api.communes(region)` (la comuna depende de la región).
   - **Campos alineados al modelo**: dirección + número separados; se quita "venue" (la API
     no lo tiene). Fechas con `input type="date"` / `time`.
   - **Submit**: al "Publicar evento" arma el `CreateEventInput` y llama `api.createEvent`;
     el evento queda `isApproved=false`. Pantalla de éxito al terminar; errores inline.
   - Se quita el texto de "el pago se procesa por nuestra pasarela" (events-only).

## Decisiones

- Imágenes (banner/poster/galería) se difieren al plan 03-02 — el evento se crea sin
  imágenes por ahora.
- Se elimina el campo "venue" del formulario: el modelo `Event` no lo tiene.
- Tras publicar se muestra una pantalla de éxito (no se redirige), avisando que el evento
  queda en revisión.

## Verificación

- `pnpm build` del website compila sin errores
- Sin sesión, `/crear` redirige a `/login`
- Con sesión, completar el formulario y publicar crea el evento vía `POST /events`
  (`isApproved=false`); aparece en `GET /events/admin` y NO en el listado público
- Errores de validación de la API se muestran en el formulario
