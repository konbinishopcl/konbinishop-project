# Phase 3 · Plan 03-03: Panel del organizador (/cuenta)

**One-liner:** Conectar `/cuenta` a `GET /events/mine` para que el organizador vea sus
eventos con su estado de moderación — cierre de Phase 3.

## Contexto

`/cuenta` lista publicaciones mock (`PUBS`). La API tiene `GET /events/mine` (Phase 1) y el
website ya tiene `api.myEvents(token)`. Este plan lo conecta — el organizador ve lo que creó
desde `/crear` y en qué estado está.

## Tareas

1. **`/cuenta/page.tsx`**:
   - **Auth gate**: sin sesión → `/login`.
   - Trae `api.myEvents(token)`; cada evento deriva su estado: `isRejected` → Rechazado,
     `isApproved` → Publicado, si no → En revisión.
   - Tabs: Todos / En revisión / Publicados / Rechazados (se quita "Archivados", la API no
     tiene ese estado) con sus conteos.
   - Cada card: imagen real, badge de estado, título, fecha, lugar, precio; el motivo de
     rechazo se muestra en los eventos rechazados; "Ver publicación" enlaza a `/evento/<slug>`
     solo para los aprobados.
   - Sidebar con los datos reales del usuario; el botón "Cerrar sesión" hace `logout()`.

## Decisiones

- Estado derivado de `isApproved` / `isRejected` (la API no expone un enum de estado).
- Sin edición de eventos en este plan (no está en el alcance de Phase 3) — solo ver.
- "Ver publicación" solo para eventos aprobados (el detalle público da 404 si no lo está).

## Verificación

- `pnpm build` del website compila sin errores
- Sin sesión, `/cuenta` redirige a `/login`
- Con sesión, `/cuenta` lista los eventos del organizador con el estado correcto
- Cierra Phase 3: publicar (crear + imágenes) y ver lo publicado funciona de punta a punta
