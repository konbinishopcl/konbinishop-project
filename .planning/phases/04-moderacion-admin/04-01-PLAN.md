# Phase 4 · Plan 04-01: Moderación de eventos en el panel admin

**One-liner:** Conectar `/dashboard/events` a la API — listar todos los eventos y aprobar /
rechazar contra `PATCH /events/:id/approve` y `/reject`.

## Contexto

`/dashboard/events` lista eventos mock (`admin-data.ts`) con columnas de ventas (sold/stock)
que no aplican al modelo events-only. La API tiene `GET /events/admin`, `PATCH
/events/:id/approve` y `/reject` (Phase 1). Este plan conecta la moderación real.

## Tareas

1. **`lib/api.ts`** — `api.adminEvents(token)` (`GET /events/admin`),
   `api.approveEvent(id, token)`, `api.rejectEvent(id, reason, token)`; el tipo `ApiEvent`
   gana `owner` (lo incluye la respuesta de admin).
2. **`/dashboard/events/page.tsx`** — reescribir: trae `GET /events/admin`, tabla de eventos
   reales (evento, organizador, fecha/lugar, precio, estado) con filtro por estado y
   búsqueda; acciones **Aprobar** y **Rechazar** (con motivo) por fila para los eventos en
   revisión. Se quitan las columnas de ventas, "destacado", export CSV y acciones masivas
   (no aplican al modelo).

## Decisiones

- Estado derivado de `isApproved` / `isRejected`.
- Rechazar pide el motivo con un `prompt()` (mínimo 3 caracteres, lo exige la API).
- Tras aprobar/rechazar se actualiza el estado en memoria (no se recarga toda la lista).
- Sin columnas de ventas ni "destacado": el modelo `Event` no los tiene.

## Verificación

- `pnpm build` del website compila sin errores
- `/dashboard/events` lista los eventos reales con su estado
- Aprobar un evento en revisión lo deja `isApproved` (visible en el sitio público)
- Rechazar con motivo lo deja `isRejected` con el motivo guardado
