# Phase 5 · Plan 05-01: Búsqueda de eventos

**One-liner:** Crear la página `/busqueda` con búsqueda por texto + filtros de categoría y
región (sincronizados con la URL) y conectar el botón de búsqueda del `Header`.

## Contexto

`GET /events` ya acepta `q`, `category` y `region` (Phase 1), y `api.events(query)` ya los
soporta. Falta la UI: una página de resultados y el acceso desde el header. El rango de
fechas requiere tocar la API y queda para un plan posterior (05-02).

## Tareas

1. **`app/(site)/busqueda/page.tsx`** — server component mínimo que envuelve en `<Suspense>`
   un componente cliente `SearchView`.
2. **`app/(site)/busqueda/SearchView.tsx`** — client component:
   - Lee `q`, `category`, `region` de la URL (`useSearchParams`).
   - Input de búsqueda + selects de categoría y región (de `api.categories()` /
     `api.regions()`); al cambiar, hace `router.push` a `/busqueda?...` (filtros shareables).
   - Trae `api.events({ q, category, region })` y muestra los resultados con `EventCard`;
     estado de carga y estado vacío.
3. **`Header.tsx`** — el botón de búsqueda navega a `/busqueda`.

## Decisiones

- `page.tsx` envuelve `SearchView` en `<Suspense>` (requisito de `useSearchParams` en
  Next 15) y marca la ruta `dynamic`.
- Filtros sincronizados con la URL → resultados compartibles (SRCH-04).
- Se reutiliza `EventCard` y la API existente; sin cambios en `lib/api.ts`.
- Rango de fechas: fuera de este plan (requiere `desde`/`hasta` en la API) → 05-02.

## Verificación

- `pnpm build` del website compila sin errores
- `/busqueda?q=anime` lista eventos cuyo título/descripción contienen el texto
- Filtrar por categoría y región acota los resultados; la URL refleja los filtros
- El botón de búsqueda del header lleva a `/busqueda`
- Sin resultados → estado vacío
