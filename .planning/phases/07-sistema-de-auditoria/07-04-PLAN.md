auditoría: CREATE y UPDATE (por organizadores y admins), y APPROVE / REJECT / BAN / DELETE.

capturar IP / userAgent / url.

respecto de la investigación: se auditan también CREATE y UPDATE de eventos por organizadores
(no solo las acciones de admin).

Output: `EventsService` con 6 puntos de auditoría, `EventsController` pasando `req`, y
`EventsModule` importando `AuditModule`.
</objective>

  <action>
    Modificar `EventsService` para registrar auditoría en cada mutación.

    1. Inyectar `AuditService` en el constructor (agregar
       `private readonly audit: AuditService` y el import
       `import { AuditService } from '../audit/audit.service';`). Importar también
       `import type { Request } from 'express';`.

    2. Agregar un parámetro opcional `req?: Request` al FINAL de la firma de cada método de
       mutación: `create`, `update`, `remove`, `approve`, `reject`, `ban`.

    3. Después de cada operación de escritura exitosa (justo después del
       `prisma.event.create/update/delete`, antes del envío de mail y del `return`), llamar a
       `this.audit.log({...})`:

       - `create`: tras crear el evento →
         `this.audit.log({ userId: user.sub, action: 'CREATE', entity: 'EVENT', entityId: event.id, req });`
         (capturar el resultado del create en una variable `event`).
       - `update`: tras actualizar →
         `this.audit.log({ userId: user.sub, action: 'UPDATE', entity: 'EVENT', entityId: id, req });`
       - `remove`: tras `prisma.event.delete` →
         `this.audit.log({ userId: user.sub, action: 'DELETE', entity: 'EVENT', entityId: id, req });`
       - `approve`: tras actualizar →
         `this.audit.log({ userId: user.sub, action: 'APPROVE', entity: 'EVENT', entityId: id, req });`
       - `reject`: tras actualizar →
         `this.audit.log({ userId: user.sub, action: 'REJECT', entity: 'EVENT', entityId: id, metadata: { reason }, req });`
       - `ban`: tras actualizar →
         `this.audit.log({ userId: user.sub, action: 'BAN', entity: 'EVENT', entityId: id, metadata: { reason }, req });`

    4. `log()` es fire-and-forget (void, no async) — NO usar `await`. Llamarlo simplemente
       como statement. Colocarlo antes del bloque de envío de mail para que un fallo de mail
       no impida el registro.

    REGLAS:
    - NO incluir `dto` completo ni `req.body` en `metadata` — solo el `reason` donde aplica
      (evita PII / datos sensibles).
    - NO hacer que un fallo de auditoría revierta la operación — `log()` ya se traga sus
      errores internamente; basta con llamarlo sin await.
    - NO modificar la lógica de negocio existente (slug, mails, includes).
  </action>

    - Agregar `import type { Request } from 'express';`.
    - En los handlers `create`, `update`, `remove`, `approve`, `reject`, `ban`: agregar el

      Ejemplos exactos del cambio:
      ```ts

        return this.events.create(dto, user, req);

        return this.events.approve(id, user, req);

      reject(@Param('id', ParseIntPipe) id: number, @Body() dto: RejectEventDto, @CurrentUser() user: JwtUser, @Req() req: Request) {

      }
      ```
