# Phase 8 Plan 03: Organizations Schema (SCH-03) Summary

Schema additions enabling the GitHub-style membership model: enum OrgRole (OWNER/MEMBER) + explicit join table OrgMember + OrgInvitation with token/expiry; inverse relations on User declared; migration sch03_organizations applied.

## What Was Built

### Models Created

**OrgMember** ŌĆö explicit many-to-many join table between a PERSON user (`userId`) and an ORGANIZATION user (`orgId`). Carries the `role` attribute (OrgRole enum). Uses two named self-referential relations on User (`UserMemberships` / `OrgMembers`) to satisfy Prisma's validation of both sides. Constraints: `@@unique([userId, orgId])` prevents duplicate memberships; cascade deletes on both FK sides.

**OrgInvitation** ŌĆö invitation record with a unique `token`, `email`, `orgId` FK, and `expiresAt`. Token-based flow: generated by OWNER, accepted by the invited user. Cascade delete when the org is deleted.

### Enum Added

**OrgRole** ŌĆö two values: `OWNER` (full control of org + members) and `MEMBER` (operate on behalf of org, shared cart). Default on OrgMember is `MEMBER`.

### Inverse Relations Added to User

```
memberships    OrgMember[]     @relation("UserMemberships")  // orgs this user belongs to
orgMembers     OrgMember[]     @relation("OrgMembers")        // members of this user if ORGANIZATION
orgInvitations OrgInvitation[] @relation("OrgInvitations")    // invitations issued by this ORGANIZATION user
```

These are required by Prisma for named self-referential relations ŌĆö without them `prisma validate` fails.

### Migration

**`20260524234414_sch03_organizations`** ŌĆö creates `OrgMember` and `OrgInvitation` tables. Purely additive; no existing table altered at the SQL level (User inverse relations are Prisma-layer metadata only).

## Pattern Applied

Explicit join table with an attribute column ŌĆö the same pattern as `OrderItem` (join table with `type`, `days`, `unitPrice`). This gives Phase 9 a concrete model to query (`prisma.orgMember.findMany({ where: { userId } })`) instead of an implicit M:M that hides the role.

Self-referential named relation pattern: both `OrgMember.userId` and `OrgMember.orgId` point to `User.id` using distinct relation names (`UserMemberships` vs `OrgMembers`). Prisma requires each named relation to have its inverse declared on the referenced model.

## Decisions Made (Locked)

1. **Service-layer enforcement of `type === ORGANIZATION` for `orgId`**: MySQL does not support CHECK constraints that cross row boundaries (i.e., you cannot enforce `User.type = 'ORGANIZATION'` on a FK). Phase 9 OrganizationsService must validate this at the application level before creating an OrgMember or OrgInvitation.

2. **`orgMembers` (not `orgMemberships`) as field name on User**: The acceptance criteria greps use `orgMembers`. The plan's `must_haves.truths` prose mentions `orgMemberships` ŌĆö this is a typo/inconsistency in the plan; the action snippets and grep patterns are authoritative.
