<objective>
Integrar el contexto de organización (`OrgContextGuard` + `@OrgContext()` de 09-02) en los 4 módulos existentes que crean/consultan recursos con ownership: **events**, **spots**, **heroes**, **orders**.

Cambio de contrato: cada `create()` y `findMine()` en estos services acepta un parámetro opcional `orgContext?: OrgContextDto | null`. Cuando está presente, el ownership del recurso (`userId` en Event/Spot/Hero, `orgId` en Order) se asigna a `orgContext.orgId` en vez del `user.sub` del caller. El listado mine filtra equivalentemente.

Crítico para Orders (Pitfall #6 del schema): `Order.userId` sigue siendo el user humano (siempre presente), pero `Order.orgId` se asigna en MODO ORG. El carrito se identifica por la tupla `(userId, orgId, status=DRAFT)`. Antes de asignar `orgId`, validar que el target tiene `type === 'ORGANIZATION'`.

Purpose: ORG-05. Cierra el ciclo de contexto de org — sin este plan, crear contenido bajo el header `X-Org-Context` no tiene efecto real. Es el plan que conecta el "X-Org-Context" abstracto del 09-02 con la persistencia de datos.

Output: Los 4 services aceptan orgContext, los 4 controllers propagan el contexto resuelto por el guard, el carrito se vuelve "uno por contexto", y la API valida `type=ORGANIZATION` antes de aceptar el orgId en Order.
</objective>

From apps/api/src/common/org-context/:
```typescript
export type OrgContextDto = { orgId: number; role: OrgRole };

export const OrgContext: ParameterDecorator;  // devuelve OrgContextDto | null
```

Patrón a aplicar en cada controller (idéntico en events/spots/heroes/orders):

```typescript
@Post()
@UseGuards(JwtAuthGuard, OrgContextGuard)
create(@Body() dto, @CurrentUser() user: JwtUser, @OrgContext() ctx: OrgContextDto | null, @Req() req: Request) {
  return this.service.create(dto, user, ctx, req);
}
```

Patrón a aplicar en cada service:
```typescript
create(dto, user, orgContext: OrgContextDto | null = null, req?: Request) {
  const ownerId = orgContext?.orgId ?? user.sub;
  return this.prisma.event.create({ data: { ..., owner: { connect: { id: ownerId } } } });
}

findMine(user, orgContext: OrgContextDto | null = null) {
  const ownerId = orgContext?.orgId ?? user.sub;
  return this.prisma.event.findMany({ where: { userId: ownerId }, ... });
}
```

From apps/api/prisma/schema.prisma (Order es DIFERENTE — tiene userId Y orgId separados):
```prisma
model Order {
  userId Int           // user humano que actuó (siempre requerido)
  orgId  Int?          // org context (opcional, FK a User type=ORGANIZATION)
  // Pitfall #6: MySQL no puede CHECK cross-row; validar en service layer
}
```

Por lo tanto en Orders, el patrón es:
```typescript
const where = { userId: user.sub, orgId: orgContext?.orgId ?? null, status: 'DRAFT' };
// carrito personal: orgId = null; carrito de org: orgId = X
```

NOTA: Es necesario importar `OrgContextGuard` desde `'../common/org-context/org-context.guard'` y agregar `OrgContextGuard` a `providers` del módulo no es necesario (es global desde 09-02). Solo importar para `@UseGuards`.
