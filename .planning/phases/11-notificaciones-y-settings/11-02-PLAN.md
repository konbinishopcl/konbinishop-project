```typescript
create(params: {
  type: NotificationType;
```

NotificationType enum (apps/api/prisma/schema.prisma:602-622) — los 19 valores; los
relevantes para CFG-02 son:
- EVENT_APPROVED / EVENT_REJECTED / EVENT_BANNED
- SPOT_APPROVED / SPOT_REJECTED / SPOT_BANNED
- HERO_APPROVED / HERO_REJECTED / HERO_BANNED
- ORG_INVITATION
- TRANSFER_REQUEST / TRANSFER_ACCEPTED / TRANSFER_REJECTED

(ARTICLE_APPROVED/REJECTED/BANNED y SUBSCRIPTION_* quedan diferidos: ARTICLE a Phase 13,
SUBSCRIPTION a Phase 12. Este plan NO los toca.)

Patrón de service injection — apps/api/src/spots/spots.module.ts:
```typescript
@Module({
  imports: [AuthModule, AuditModule], // se agrega NotificationsModule aquí
  controllers: [SpotsController],
  providers: [SpotsService],
})
```

User model (apps/api/prisma/schema.prisma):
- `User.type: UserType` enum con valores `PERSON | ORGANIZATION` (SCH-01)
- Items con dueño (Event/Spot/Hero/Article) tienen `userId` que apunta a un User
- Cuando el dueño es una org, `userId` apunta a un User con `type=ORGANIZATION`
  → ese userId ES el orgId para fines de notificación

Owner.type lookup pattern: los services ya incluyen `owner: { select: { email, firstname } }`
en approve/reject/ban — extender el select a `{ id, email, firstname, type }` para
resolver el recipient en cada caso.

OrgMember.role enum: `OWNER | MEMBER` (de SCH-03).

    **Regla de recipient (única, aplicada en los 3 services):**
    El dueño de un item (Event/Spot/Hero) está en `userId`. Cuando el dueño es una
    organización, ese userId apunta a un `User` con `type=UserType.ORGANIZATION`. La
    regla de notificación es:
    ```typescript
    // Helper inline (duplicar en cada service — son 6 líneas, no vale la pena un módulo común)
    function recipient(owner: { id: number; type: UserType }) {
      return owner.type === UserType.ORGANIZATION
        ? { orgId: owner.id }
        : { userId: owner.id };
    }
    ```

    **Para cada uno de Events/Spots/Heroes:**
