<objective>
Build the typed API foundation that Plans 03 (CreateProductView), 04 (UpsellView),
and 05 (dashboard sections) all consume. Today `lib/api.ts` has NO `ApiSpot` type
at all, an INCOMPLETE `ApiHero` (missing status/statusReason/owner/userId/createdAt),
and no spot/hero methods beyond `heroes()`.

This plan:
1. Adds the `ApiSpot` type and completes `ApiHero`.
2. Adds `ApiQuota` type.
3. Adds all spot + hero API methods (create, update, delete, approve, reject, ban,
   mine, quota, adminSpots, adminHeroes).
4. Updates `api.heroes()` to the new paginated response shape from Plan 01.
5. Adds the `.field-error` CSS class to globals.css (mandated by phase goal —
   per-field Zod errors render as `<p className="field-error">`). This file is owned
   HERE so Plans 03/04 (parallel in Wave 2) never both edit globals.css.

NOTE on `orgId`: the phase goal listed `orgId` as a field to add to ApiHero. The
Prisma schema (verified) has NO orgId column on Spot or Hero — ownership is a single
`userId` (org context is resolved at create time on the server). So `orgId` is
intentionally OMITTED from the types. Document this in the SUMMARY.

Purpose: One typed source of truth for spots/heroes so the three downstream plans
don't hand-roll fetch shapes.
Output: extended `lib/api.ts`, `.field-error` in globals.css.

Requirements (defined inline for this phase):
- **SPOT-07/08/09/10:** Typed approve/reject/ban/quota+admin-list methods exist for spots.
- **HERO-06/07/08:** Typed approve/reject/ban methods exist for heroes.
</objective>
