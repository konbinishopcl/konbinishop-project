<objective>
Eliminar el endpoint backend `PATCH /users/me/organizer` y todo su soporte: el método `updateOrganizer` del controller y del service, y el DTO `UpdateOrganizerDto` con sus imports.

Purpose: Este endpoint solo lo consumía la página `/cuenta/organizador`, que se elimina en el Plan 01. Con el modelo post-Phase-22, el concepto de "perfil de organizador editable desde el usuario" desaparece (los campos bio/website del Profile se moverán a org settings en una fase futura). Mantener el endpoint dejaría código muerto y una superficie de API sin uso.
Output: Backend de users sin rastro del endpoint organizer; compila y arranca limpio.

**Por qué Wave 2 (depends_on 01 y 02):** El único caller frontend de `/api/users/me/organizer` es `cuenta/organizador/page.tsx`, eliminado por Plan 01. Esperar a Wave 2 garantiza que no exista una ventana donde el frontend llame a un endpoint ya borrado. Plan 02 corre en paralelo a 01 (Wave 1) y no toca el backend; se incluye en depends_on para que la verificación de Wave 2 ocurra con todo el frontend ya consolidado.
</objective>
