# Phase 27.1: Dashboard Gap Fixes — Context

**Gathered:** 2026-05-30
**Status:** Ready for planning
**Source:** Manual audit of dashboard sections post-phases 25-27

<domain>
## Phase Boundary

This phase eliminates all remaining mock data and broken interactions in the admin dashboard after phases 25-27. Those phases intentionally deferred items labeled "no aggregate endpoint" or "KPIs stay mock". This phase resolves them using client-side computation from already-loaded data or cheap extra API calls to existing endpoints.

**In scope:**
- HomeSection: real KPIs (computed from existing APIs), real category bars, "Ver toda →" navigation
- PaymentsSection: KPIs from loaded data, column rename, remove fake buttons
- ReportsSection: top-organizadores from real payments data (client-side grouping)
- SubsSection: load plan config from /api/settings on mount
- DashboardShell: fix breadcrumb to show group name instead of always "DASHBOARD"
- Países/Divisiones/Ciudades (SimpleCRUDSection): connect to real /countries, /states, /cities API
- EventsSection + ArticlesSection approve modal: remove fake AI tag generator

**Out of scope:**
- KPI deltas (↑ 24%, etc.) — requires historical aggregate endpoints, not buildable client-side
- "Descargar comprobante" PDF — no backend PDF generation
- "Reembolsar" — no refund endpoint in backend
- Photography/content-creators inbox — intentionally redirects to CRM
- Top-eventos por organizador in ReportsSection — no events-by-owner aggregate; remove or keep placeholder

</domain>

## Implementation Decisions

### DASH-FIX-01: HomeSection real KPIs

Three KPIs can be made real with existing endpoints:

1. **EVENTOS PUBLICADOS** — call `api.adminEvents(token, { status: "APPROVED", pageSize: 1 })` (fetches minimal data, uses `total` field). No need to load all events.
2. **SUSCRIPTORES** — call `api.subscriptions(token)` and use `res.total`.
3. **INGRESOS MES** — from already-loaded payments (HomeSection should fetch payments for the current month via `api.adminPayments(token)` filtered client-side to current month).

The fourth KPI (INGRESOS MES) should use the same `api.adminPayments(token)` call that ReportsSection uses. Since HomeSection runs independently, add a small payments fetch on mount.

KPI deltas (↑ 24%, ↑ 18% etc.) — **remove entirely** rather than keep fake numbers. Empty delta slot is acceptable.

### DASH-FIX-02: HomeSection category bars

Fetch `GET /event-categories` (public endpoint, `api.categories()` method exists but returns event categories only). The response includes `_count.events` IF requested — need to check; if not, call with explicit include.

Actually `GET /event-categories` is the endpoint and the backend returns the category list. Check if `_count` is included. The EventCategoriesSection type shows `(c as ApiEventCategory & { _count?: { events?: number } })._count?.events ?? "—"` — so the API DOES return `_count.events`.

**Approach:** Fetch `/event-categories` (which returns ApiEventCategory[] with `_count.events`), compute bar percentages from `_count.events / totalEvents * 100`. Sort by count descending, show top 6.

### DASH-FIX-03: HomeSection "Ver toda →"

Change the `<button className="sel">Ver toda →</button>` to `<Link href="/dashboard/events" className="sel" style={...}>Ver toda →</Link>`.

### DASH-FIX-05: ReportsSection top-organizadores from real data

**TOP_INGRESOS (by buyer from loaded payments):**
```ts
const topIngresos = useMemo(() => {
  const map = new Map<string, number>();
  payments.filter(p => p.status === "PAID").forEach(p => {
    map.set(p.buyer.name, (map.get(p.buyer.name) ?? 0) + p.total);
  });
  return [...map.entries()]
    .sort((a,b) => b[1]-a[1])
    .slice(0, 5)
    .map(([name, total]) => [name, "$"+total.toLocaleString("es-CL")] as [string,string]);
}, [payments]);
```

Bar widths: `width = (value / maxValue * 100) + "%"` where `maxValue = topIngresos[0]?.[1]`.

**TOP_EVENTOS:** Remove the hardcoded list entirely. Replace panel with a note "Próximamente — requiere agregado por organizador" OR remove the panel completely. Decision: **remove the panel**, keep only TOP_INGRESOS. The two-column layout becomes single-column or can be replaced with a simple table of buyer totals.

### DASH-FIX-06: SubsSection loads plan config on mount

Pattern identical to SettingsSection (which does fetch("/api/settings") and patches state). 

On mount, SubsSection should:
```ts
useEffect(() => {
  if (!token) return;
  fetch("/api/settings", { headers: { Authorization: `Bearer ${token}` } })
    .then(r => r.ok ? r.json() : null)
    .then(data => {
      if (!Array.isArray(data)) return;
      const keys: (keyof PlanSettings)[] = ["sub_monthly_price","sub_credits_per_month","sub_aviso_discount","sub_portada_discount"];
      const patch: Partial<PlanSettings> = {};
      data.forEach(({ key, value }) => { if (keys.includes(key as keyof PlanSettings)) patch[key as keyof PlanSettings] = value; });
      if (Object.keys(patch).length) setPlan(p => ({ ...p, ...patch }));
    })
    .catch(() => {});
}, [token]);
```

### DASH-FIX-08: Catálogos geo con API real

**Current state:** SimpleCRUDSection with `kind="countries"|"states"|"cities"` uses hardcoded mock data and fake CRUD (only toast.success, no API calls).

**Backend endpoints (confirmed):**
- `GET/POST/PATCH/:id/DELETE/:id /countries` — Country: { id, name, slug }
- `GET/POST/PATCH/:id/DELETE/:id /states` — State: { id, name, slug, countryId, country? }  
- `GET/POST/PATCH/:id/DELETE/:id /cities` — City: { id, name, slug, stateId, state? }

**Approach:** Create three dedicated section components (or one parametric `GeoCRUDSection`) replacing the mock data in SimpleCRUDSection.

Add API methods to `lib/api.ts`:
```ts
// Countries admin CRUD
adminCountries: () => request<ApiCountry[]>("/countries"),
createCountry: (body: { name: string; slug: string }, token: string) => request<ApiCountry>("/countries", { method: "POST", body: JSON.stringify(body) }, token),
updateCountry: (id: number, body: { name?: string; slug?: string }, token: string) => request<ApiCountry>(`/countries/${id}`, { method: "PATCH", body: JSON.stringify(body) }, token),
deleteCountry: (id: number, token: string) => request<void>(`/countries/${id}`, { method: "DELETE" }, token),

// States admin CRUD
adminStates: (country?: string) => request<ApiRegion[]>(`/states${country ? `?country=${encodeURIComponent(country)}` : ""}`),
createState: (body: { name: string; slug: string; countryId: number }, token: string) => request<ApiRegion>("/states", { method: "POST", body: JSON.stringify(body) }, token),
updateState: (id: number, body: { name?: string; slug?: string }, token: string) => request<ApiRegion>(`/states/${id}`, { method: "PATCH", body: JSON.stringify(body) }, token),
deleteState: (id: number, token: string) => request<void>(`/states/${id}`, { method: "DELETE" }, token),

// Cities admin CRUD
adminCities: (state?: string) => request<ApiCommune[]>(`/cities${state ? `?state=${encodeURIComponent(state)}` : ""}`),
createCity: (body: { name: string; slug: string; stateId: number }, token: string) => request<ApiCommune>("/cities", { method: "POST", body: JSON.stringify(body) }, token),
updateCity: (id: number, body: { name?: string; slug?: string }, token: string) => request<ApiCommune>(`/cities/${id}`, { method: "PATCH", body: JSON.stringify(body) }, token),
deleteCity: (id: number, token: string) => request<void>(`/cities/${id}`, { method: "DELETE" }, token),
```

**Column structure (no iso/flag — they don't exist in schema):**
- Countries: NOMBRE | SLUG | DIVISIONES (count from states) | actions
- States: NOMBRE | SLUG | PAÍS | CIUDADES (count) | actions
- Cities: NOMBRE | SLUG | DIVISIÓN | actions

The page.tsx files for `/dashboard/countries`, `/dashboard/states`, `/dashboard/cities` currently import `SimpleCRUDSection kind="countries"|"states"|"cities"`. Replace those page imports with new dedicated section components, or update SimpleCRUDSection to detect kind and fetch real data.

**Simplest approach:** Convert the `countries`, `states`, `cities` branches of SimpleCRUDSection to fetch from real API (mirroring the existing `tags` branch which already does `RealTagsSection`). Add `RealCountriesSection`, `RealStatesSection`, `RealCitiesSection` inside `SimpleCRUDSection.tsx`.

### EventsSection + ArticlesSection approve modal cleanup

**Current problem:** `AdminApproveModal` (inline in EventsSection and ArticlesSection) has:
1. Hardcoded initial tags: `useState("anime, cosplay, santiago, evento")`
2. Fake `regenAI()` with `setTimeout + Math.random()`  
3. Tags never passed to `doApprove()` — they're ignored

**Fix:** Remove the tags field and AI button from the approve modal entirely. The approve modal should just confirm with a simple "¿Publicar este evento/artículo?" message. Keep the reject modal as-is (it's functional).

The inline `AdminApproveModal` in EventsSection and ArticlesSection can be replaced by the proper `AdminApproveModal` from `/modals/AdminApproveModal.tsx` — but that modal also has the fake AI feature. 

**Decision:** Simplify the approve modal to just a ConfirmDialog (title + message + confirm button). Events and Articles don't have a tags-on-approve flow in the backend.
