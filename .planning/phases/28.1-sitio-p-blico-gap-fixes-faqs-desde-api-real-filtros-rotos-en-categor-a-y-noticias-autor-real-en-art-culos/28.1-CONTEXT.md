# Phase 28.1: Sitio público gap fixes - Context

**Gathered:** 2026-05-30
**Status:** Ready for planning

<domain>
## Phase Boundary

Eliminar datos mock y funcionalidades falsas del sitio público: (1) filtros que muestran UI pero nunca filtran datos reales, (2) FAQs hardcodeadas en frontend cuando existe `GET /faq` público, (3) autor de artículo hardcodeado cuando la relación `user` existe en el backend. Ningún cambio al diseño visual — solo conectar datos reales y eliminar código muerto.

</domain>

<decisions>
## Implementation Decisions

### Filtros rotos — eliminación limpia

- **Ciudad en `/categoria/[cat]`**: Eliminar el botón de filtro "Ciudad", el estado `filterCity`, y el contador de filtros activos que lo incluye. El array `CITY_OPTIONS` se borra. El filtro de precio, formato, fecha y orden se conservan (todos funcionan).
- **Tipo en `/noticias/categoria/[slug]`** (`NewsCategoryView.tsx`): Eliminar el chip "Tipo" (Artículos/Entrevistas/Reseñas), el estado `type`, la opción de `TYPE_OPTIONS` y su referencia en el contador de filtros activos. El filtro "Origen" (Editoriales/Patrocinados) se conserva — funciona correctamente contra `isSponsored`.
- **Tipo en `/noticias/tags/[slug]`** (`TagArticlesView.tsx`): Mismo tratamiento que NewsCategoryView — eliminar TYPE_OPTIONS y estado `type`.

### FAQs desde API real

- `/preguntas-frecuentes/page.tsx`: Convertir a `async` page, llamar `api.faqAll()` con `.catch(() => FAQS_FALLBACK)` donde `FAQS_FALLBACK` es el array hardcodeado actual.
- `/precios/PricingView.tsx` → `precios/page.tsx`: Fetch `api.faqAll()` en el server component y pasarlo como prop a PricingView. Fallback al array hardcodeado actual.
- El fallback es lógico: las preguntas hardcodeadas son el contenido real actual; si la API cae, el usuario igual ve respuestas relevantes.

### Autor real en artículos

- **Backend** (`apps/api/src/articles/articles.service.ts`): Agregar `user: { select: { firstname: true, lastname: true } }` a `ARTICLE_DETAIL_INCLUDE`. Este include ya existe en `ARTICLE_INCLUDE` (lista) pero falta en el detalle.
- **Frontend type** (`apps/website/lib/api.ts`): Extender `ApiArticle` con campo opcional `user: { firstname: string; lastname: string } | null`.
- **ArticleView** (`apps/website/app/(site)/noticias/[slug]/ArticleView.tsx`): Reemplazar `const authorName = "Konbini Editorial"` con `const authorName = article.user ? \`${article.user.firstname} ${article.user.lastname}\`.trim() : "Konbini Editorial"`. Fallback a "Konbini Editorial" cubre artículos importados donde `userId` es null.

### Claude's Discretion

- Orden de los archivos afectados, naming de constantes de fallback.
- El campo `user` en `ApiArticle` puede ser nullable o undefined — usar `user?: ...` para compatibilidad con artículos existentes sin ese campo.

</decisions>

<specifics>
## Specific Ideas

- El fallback de FAQs debe ser el array hardcodeado ACTUAL (es el contenido real del sitio) — no un array vacío.
- Para el autor: `article.user?.firstname + " " + article.user?.lastname` puede dar "undefined undefined" si no se guarda el nombre completo; usar template con `.trim()` y verificar que ambos campos existen antes de usarlos.
- No agregar "Tipo" al modelo Article — está fuera del scope de esta fase.

</specifics>
