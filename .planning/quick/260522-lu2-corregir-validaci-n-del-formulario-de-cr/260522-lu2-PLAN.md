    reutilizables (funciones puras, fuera del componente o como const al inicio del módulo).

    **Helpers a crear:**
    - `isValidUrl(value: string): boolean` — usa `new URL()` dentro de try/catch; acepta solo
      protocolos `http:`/`https:`. Devuelve `false` para string vacío.
    - `todayISO(): string` — fecha de hoy en formato `YYYY-MM-DD` para usar como `min` de los
      inputs de fecha.
    - Constante `MIN_PRICE = 500` (CLP) y `MIN_TITLE_LEN = 3`.

      `type="number"`.
    - En el `onChange` del precio: rechazar caracteres no numéricos enteros — permitir solo
      `""` o dígitos (`/^\d*$/`); ignorar el cambio si no matchea (evita pegar texto/decimales).

    `inputMode="url"` y `type="text"` (sigue siendo text por el prefijo). NO poner

    **Inputs `videos`** (Step3): cambiar a `type="url"` e `inputMode="url"` (aquí el usuario sí
    escribe la URL completa, incluido el esquema).

    selector de fecha nativo no permita fechas pasadas.

    **Input título** (`Step1`): añadir `minLength={MIN_TITLE_LEN}` y `maxLength={120}`.
    **Inputs de dirección/número** (`Step2`): añadir `maxLength` razonable (ej. 120 y 12).

<task type="auto">
  <name>Task 2: Ampliar validateStep() con reglas de negocio</name>
  <files>apps/website/app/crear/page.tsx</files>
  <action>
    Ampliar la función `validateStep(s: number)` para que bloquee el avance de paso (y el
    submit) ante datos inválidos. Mantener el patrón actual: devuelve un `FieldErrors`
    (`Record<string,string>`); las claves siguen el formato existente (`title`, `desc`,
    `price_${i}`, etc.). Los `<FieldError>` ya están cableados a esas claves.

    **Paso 1 — añadir/ajustar:**
    - `title`: además del trim no vacío, exigir `data.title.trim().length >= MIN_TITLE_LEN`
      → mensaje: "El título debe tener al menos 3 caracteres."
    - Precios (solo cuando `!data.free`), por cada `prices[i]`:
      - Nombre vacío → `price_${i}` (ya existe).
      - Monto: debe ser un entero válido. Parsear con `Number(p.amount)`; si `p.amount` está
        vacío, no es entero (`!Number.isInteger(n)`) o es `< MIN_PRICE` →
        `price_amount_${i}` = "El precio mínimo es $500 CLP." (o "Ingresa un monto válido."
        si no es número). Usar una clave distinta del nombre para no pisar el error de nombre.
    - Si TODO el formulario es de pago pero no hay ninguna tarifa válida, igual basta con la
      validación por fila anterior.

    **Paso 2 — añadir:**
    - Fechas: por cada `dates[i]` con `date` no vacío, validar que `date >= todayISO()`
      (comparación lexicográfica de strings `YYYY-MM-DD` es válida) → clave `date_${i}` =
      "La fecha no puede estar en el pasado." Exigir además que al menos haya una fecha con
      `date` no vacío → clave `dates` = "Agrega al menos una fecha para el evento."
    - Validar que si `start` y `end` están ambos presentes en una fila, `end > start`
      → `date_${i}` = "La hora de término debe ser posterior al inicio."
    - URL ticketera (`web`): si `data.web.trim()` no está vacío, validar
      `isValidUrl(\`https://\${data.web.trim()}\`)` → clave `web` = "Ingresa una dirección web
      válida." (campo opcional: vacío es válido).
    - URLs de redes (`socials`): por cada valor no vacío, normalizar quitando un `@` inicial
      si lo trae, y validar como URL (`isValidUrl` sobre el valor, anteponiendo `https://` si
      no tiene esquema) → clave `social_${i}`.

    **Paso 3 (videos)** — `validateStep` hoy solo cubre pasos 1 y 2. Añadir caso `s === 3`:
    - Por cada `videos[i]` no vacío, validar `isValidUrl(v.trim())` → clave `video_${i}` =
      "Ingresa una URL de video válida."
    - Cablear los `<FieldError>` correspondientes en `Step3` (videos) y en `Step2`
      (`web`, `social_${i}`, `date_${i}`, `dates`) usando el prop `errors`. `Step3` hoy NO
      recibe `errors` — añadir el prop `errors: FieldErrors` a `Step3` y pasárselo desde
      `FormPage` (`<Step3 ... errors={fieldErrors} />`). El botón "Publicar" debe llamar a
      `validateStep(3)` antes de `submit()` (hoy `submit()` solo valida el paso 2).

    **Toggle gratuito:** cuando el usuario marca `free`, resetear los montos de precio a `"0"`
    para que el estado quede consistente: en el handler del checkbox (`onClick` del `ck-row`),
    al pasar a `free === true`, hacer también
    `update("prices", data.prices.map(p => ({ ...p, amount: "0" })))`. Así un evento gratuito
    nunca arrastra montos tipeados antes.

    **UX (scroll a error):** tras `setFieldErrors(errs)` cuando hay errores y se bloquea el
    avance, hacer `window.scrollTo({ top: 0, behavior: "smooth" })` o, mejor, hacer
    `scrollIntoView` al primer campo con error si es trivial. El scroll al top ya es aceptable.

    No introducir Zod ni React Hook Form. Mantener mensajes en español, tono consistente con
    los existentes.
