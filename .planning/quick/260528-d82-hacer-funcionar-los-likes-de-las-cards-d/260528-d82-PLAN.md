<objective>
Hacer funcionar correctamente los likes de las cards de noticias. El sistema POST/DELETE
ya existe pero el botón siempre inicia en estado `false`: un usuario que ya dio like vuelve
a pulsar el botón y dispara un POST que choca con la restricción `@@unique` (409 Conflict),
revirtiendo visualmente el like.

Causa raíz: no hay forma de saber si el usuario actual ya likeó un artículo. Como el SSR
usa `API_KEY` (no el token del usuario), el estado por-usuario debe resolverse en cliente.

Purpose: Que el like refleje el estado real del usuario y sea reversible sin errores.
Output: Endpoint GET de estado de like + ArticleCard que lee ese estado al montar y
reconcilia con la respuesta autoritativa del servidor.
</objective>

1. En `apps/api/src/likes/likes.service.ts`, agregar un método público `status`:
   ```ts
   async status(target: LikeTarget, targetId: number, user: JwtUser) {
     const existing = await this.prisma.like.findFirst({
       where: { userId: user.sub, [`${target}Id`]: targetId },
     });
     const count = await this.countLikes(target, targetId);
     return { liked: !!existing, likes: count };
   }
   ```
   Reutiliza el método privado `countLikes` ya existente. NO valida que el target exista
   (un GET de estado sobre id inexistente simplemente devuelve `{ liked: false, likes: 0 }`).

2. En `apps/api/src/articles/articles.controller.ts`, agregar un endpoint GET en la
   sección `// ── Likes ──` (junto a los `@Post(':id/like')` / `@Delete(':id/like')`),
   protegido con `JwtAuthGuard` (igual que los otros dos):
   ```ts
   @Get(':id/like')
   @UseGuards(JwtAuthGuard)
   @ApiBearerAuth()
   @ApiOperation({ summary: 'Estado de like del usuario actual para un artículo' })
   likeStatus(@Param('id', ParseIntPipe) id: number, @CurrentUser() user: JwtUser) {
     return this.likes.status('article', id, user);
   }
   ```
   IMPORTANTE — orden de rutas: `@Get(':slug')` (línea ~54) usa `@Param('slug')` y captura
   strings. `@Get(':id/like')` tiene un segmento extra `/like`, por lo que NO colisiona con
   `:slug` (rutas de distinta longitud). Declarar el nuevo `@Get(':id/like')` en el bloque
   de Likes al final del controller es seguro.

NO usar `OptionalJwtAuthGuard`: el frontend solo llamará este endpoint cuando haya token
(los anónimos no tienen estado por-usuario).

En `apps/website/components/ArticleCard.tsx`:

1. Importar `useEffect` junto a `useState` desde "react".

2. Tras los `useState`, agregar un efecto que cargue el estado `liked` al montar, SOLO si
   hay token (anónimos se saltan el fetch — no tienen estado):
   ```tsx
   useEffect(() => {
     if (!token) return;
     let cancelled = false;
     fetch(`/api/articles/${a.id}/like`, {
       headers: { Authorization: `Bearer ${token}` },
     })
       .then((r) => (r.ok ? r.json() : null))
       .then((data: { liked: boolean; likes: number } | null) => {
         if (cancelled || !data) return;
         setLiked(data.liked);
         setLikes(data.likes);   // reconcilia el conteo con el valor autoritativo
       })
       .catch(() => { /* silencioso: deja el estado inicial */ });
     return () => { cancelled = true; };
   }, [token, a.id]);
   ```

3. En `toggleLike`, usar la respuesta del servidor como fuente de verdad y mostrar toast en
   error. Reemplazar el bloque `try/catch` actual por:
   ```tsx
   try {
     const res = await fetch(`/api/articles/${a.id}/like`, {
       method: next ? "POST" : "DELETE",
       headers: { Authorization: `Bearer ${token}` },
     });
     if (!res.ok) throw new Error("request failed");
     const data = (await res.json()) as { liked: boolean; likes: number };
     setLiked(data.liked);
     setLikes(data.likes);
   } catch {
     // Revertir el optimistic update y avisar
     setLiked(!next);
     setLikes((c) => c + (next ? -1 : 1));
     toast.error("No se pudo actualizar el like");
   }
   ```
   El optimistic update previo (setLiked/setLikes antes del fetch) se mantiene tal cual está.
